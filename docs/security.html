Firewall & Security Protocols

Cowork implements a multi-layered Zero-Trust security architecture. No tool executes without passing through the autonomous firewall validation engine.

The Tool Firewall

The firewall is the final logic gate before any execution. It evaluates rules based on tool names, intent classification, and argument-level regex patterns.

ALLOW

Immediate background execution for safe, read-only tools.

ASK

Agent pauses and requests explicit user approval before proceeding.

BLOCK

Execution strictly denied. Agent is informed of the policy violation.

Configuring firewall.yaml

Located at ~/.cowork/firewall.yaml, this file defines the global security posture.

# Example Firewall Configuration
policy:
default_action: ask
tools:
- name: "git_push"
action: ask
description: "Prevent unverified remote code changes"
- name: "codebase_bash"
action: ask
rules:
- field: command
regex: "^rm -rf"
action: block
description: "strictly block recursive deletions"

Argument-Level Filtering

Cowork supports deep inspection of tool arguments using regular expressions. This allows you to allow a tool generally but block specific dangerous payloads.

Regex Inspection

You can target specific fields in a tool call's arguments. If the regex matches, the specified action overrides the tool's default.

Trust Tiers

Tools are grouped into tiers based on their potential impact on the system.

Tier 1: Observational (ReadOnly)

Tools like web_search, codebase_read_file, or calc. Usually set to ALLOW.

Tier 2: Mutative (Workspaces)

Tools that write files or modify local session state. Usually set to ASK or ALLOW in trusted environments.

Tier 3: Critical (System/External)

Shell execution, git mutations, or email sending. Strictly set to ASK by default.

Safety First

We recommend keeping the default_action at ask for production environments. You can always use the cowork firewall reset command to restore factory-safe defaults.
